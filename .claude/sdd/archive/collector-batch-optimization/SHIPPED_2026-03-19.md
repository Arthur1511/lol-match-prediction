## What Was Built

### Core Features

#### 1. Two-Phase Collection Architecture

**Phase 1: Discovery with Progressive Caching**
- Aggressive match ID discovery (up to 50,000 unique IDs)
- Snowball sampling across multiple iterations (default: 3)
- Progressive caching: fetch details BUT don't commit yet
- Player extraction from cached matches for next iteration

**Phase 2: Selection and Commit**
- Select best N matches from cache (default: 10,000)
- Prioritize by recency using timestamp decoding
- Single-pass commit to collection
- Accurate cache hit rate calculation

#### 2. Cache Performance Tracking

**Added in `__init__`:**
```python
self._cache_hits: int = 0
self._cache_misses: int = 0
```

**Tracked in `fetch_match_details()`:**
```python
if cached is not None:
    self._cache_hits += 1
    return cached
self._cache_misses += 1
```

**Logged in Phase 2:**
```python
total_lookups = self._cache_hits + self._cache_misses
cache_hit_rate = (self._cache_hits / total_lookups * 100)
logger.info(f"Cache hits: {self._cache_hits}, misses: {self._cache_misses}")
logger.info(f"Cache hit rate: {cache_hit_rate:.1f}%")
```

#### 4. Match ID Timestamp Decoding

**Utility Functions:**
```python
def decode_match_timestamp(match_id: str) -> int:
    """Extract timestamp from Riot match ID (e.g., BR1_1234567890_1)"""
    parts = match_id.split('_')
    return int(parts[1]) if len(parts) >= 2 else 0

def sort_match_ids_by_recency(match_ids: list) -> list:
    """Sort match IDs by recency (most recent first)"""
    return sorted(match_ids, key=decode_match_timestamp, reverse=True)
```

### Technical Decisions

#### Decision 1: Two-Phase Architecture

**Choice:** Separate discovery (Phase 1) from commitment (Phase 2)

**Rationale:**
- Enables deduplication before commitment
- Allows selecting best N matches from larger pool
- Improves cache hit rate significantly

**Alternatives Considered:**
1. Single-phase with larger cache → Still 0% hit rate
2. Random sampling from discovered matches → Lower quality data
3. Time-based partitioning → Too complex for minimal gain

**Consequence:** More complex code, but significant performance improvement

**Lesson:** Separate concerns (discovery vs selection) for better optimization

#### Decision 2: Timestamp Decoding for Sorting

**Choice:** Extract timestamp from match ID for recency sorting

**Rationale:**
- No additional API calls needed
- Match IDs encode timestamps ( Riot API design)
- Most recent matches are better for ML training

**Alternatives Considered:**
1. Random selection → Lower quality
2. Fetch all metadata first → Too slow
3. Use gameCreation timestamp → Requires API call

**Consequence:** Better ML data with minimal overhead

**Lesson:** Leverage existing data structures (match IDs) for optimization

#### Decision 3: Config Migration in `__init__`

**Choice:** Migrate legacy config keys during collector initialization

**Rationale:**
- Config always in migrated state
- Tests work without manual intervention
- Backward compatibility guaranteed

**Alternatives Considered:**
1. Migrate in `run_collection()` → Tests failed, had to move to `__init__`
2. Require users to update configs → Breaking change
3. Separate migration script → Too complex

**Consequence:** Automatic backward compatibility, no user action required

**Lesson:** Migrate configs at initialization, not at runtime

#### Decision 4: Cache Hit Rate Calculation

**Choice:** Track hits/misses during fetch operations

**Rationale:**
- Measures actual cache effectiveness
- Accurate metric for monitoring
- Simple to implement

**Initial Implementation (Wrong):**
```python
cache_hit_rate = ((len(cache) - committed) / len(cache)) * 100
# Measured leftovers, not hits
```

**Corrected Implementation:**
```python
total_lookups = self._cache_hits + self._cache_misses
cache_hit_rate = (self._cache_hits / total_lookups * 100)
# Measures actual effectiveness
```

**Consequence:** Now correctly measures cache hit rate

**Lesson:** Verify metric calculations measure what you intend to measure
