## Lessons Learned

### What Went Well

1. **User-provided context**: The `api_rate_limits.md` file with detailed per-endpoint limits was crucial for designing the solution. This highlights the value of documenting API constraints before implementation.

2. **Config-driven approach**: Making all optimizations configurable via `config.yaml` with sensible defaults enabled backward compatibility and easy rollback.

3. **Simple technical choices**: Using `OrderedDict` for LRU cache instead of external dependencies or complex decorators kept the implementation simple and async-native.

4. **Composition over inheritance**: `EndpointAwareRateLimiter` containing multiple `RateLimiter` instances (rather than extending the class) preserved the existing battle-tested code and made testing easier.

5. **Phased implementation**: Building in order (classes → integration → config → tests → docs) prevented integration issues and made debugging straightforward.

### What Could Be Improved

1. **Test execution time**: Async rate limiter tests take ~2 minutes because they actually wait for rate limits to refill. Could use shorter time windows for faster testing.

2. **Memory tracking**: No actual memory profiling was done to verify the <300MB overhead claim. Should add memory profiling to future performance tests.

3. **Cache hit rate targets**: The >20% cache hit rate target is theoretical (based on high-elo overlap). Real-world validation pending.

4. **CLAUDE.md update**: Planned but not executed due to time. Should add patterns for the new classes to help future Claude instances.

### Technical Decisions That Paid Off

1. **Substring matching for endpoint routing**: Simple and fast. Riot's API structure is stable, so no need for more complex regex or config-driven patterns.

2. **Cache only match details**: Not caching match IDs or ladder data (YAGNI) provided 80% of benefit with 20% of complexity.

3. **Feature flags via config**: All optimizations can be disabled without code changes, making rollback trivial.

### Process Improvements

1. **AgentSpec workflow**: The 5-phase workflow (brainstorm → define → design → build → ship) provided clear structure and prevented scope creep.

2. **YAGNI application**: Explicitly removing features (cache persistence, multi-region, adaptive limiting) kept the implementation focused and delivered quickly.
